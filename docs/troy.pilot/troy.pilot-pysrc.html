  2  """ 
  3   
  4  Troy is an implementation of the Pilot API, which is an interface to pilot 
  5  abstractions.  Most prominently, pilot abstractions cover PilotJobs, a well 
  6  known mechanism for application level job management, which is usually employed 
  7  for HTC type applications on HPC type resources (amongst others).  Another pilot 
  8  abstraction is PilotData, which, in a similar fashion as PilotJobs, decouples 
  9  system level and application level data management. 
 10   
 11  The Pilot API exposes the ability to handle Compute and Data workloads 
 12  concurrently, by interleaving PilotJobs and PilotData concepts. 
 13   
 14  The main concepts and classes of the API are: 
 15   
 16      - L{ComputePilot} (CP) / L{DataPilot} (DP): 
 17        a pilot, which manages a (compute or data) resource slice for the 
 18        application 
 19   
 20      - L{ComputePilotService} (CPS) / L{DataPilotService} (DPS): 
 21        creates and manages pilots, according to application specified resource 
 22        requirements.  Also, 'submit_xxx_unit()' methods accept work descriptions 
 23        of L{ComputeUnit}s and L{DataUnit}s to be executed on its pilots. 
 24   
 25      - L{ComputeUnitDescription} (CUD) / L{DataUnitDescription} (DUD) / 
 26        L{ComputeDataUnitDescription} (CDUD) : 
 27        describes a piece of (compute or data or combination thereof) workload, to 
 28        be managed by a (set of) pilot(s). 
 29   
 30      - L{ComputeUnit} (CU) / L{DataUnit} (DU) / L{ComputeDataUnit} (CDU): 
 31        represents an instance of a piece of (compute or data) workload, which was 
 32        created according to a L{ComputeUnitDescription} / L{DataUnitDescription} 
 33        / L{ComputeDataUnitDescription} 
 34   
 35      - L{ComputeUnitService} (CUS) / DataUnitService (DUS) / 
 36        ComputeDataUnitService (CDUS): 
 37        an application level scheduler which can instantiate L{ComputeUnit}s, 
 38        L{DataUnit}s and L{ComputeDataUnit}s on a set of L{ComputePilotService}s 
 39        and L{DataPilotService}s, and their pilots. 
 40   
 41  The UnitService's is what the application will mostly work with: it takes care 
 42  of proper work item distribution, and will in fact enact the applications 
 43  workload.  The set of pilots available to the UnitService can be changed during 
 44  the application's runtime.  A simple (compute) example on 'Hello World' level 
 45  would be (some details left out for brevity):: 
 46   
 47      ---------------------------------------------------------------- 
 48      import troy 
 49   
 50      # create on ComputePilot from a BigJob ComputePilotservice 
 51      cps = troy,pilot.ComputePilotService ('bigjob://') 
 52      cps.submit_pilot ([some compute pilot description]) 
 53   
 54      # create a work unit service with one ComputePilot resource, to submit work 
 55      # items to 
 56      cus = troy.pilot.ComputeUnitService () 
 57      cus.add_pilot_service (cps) 
 58   
 59      # submit a compute work item 
 60      cu = cus.submit_unit ([some compute unit description]) 
 61   
 62      cu.wait ()     # wait 'til work is done 
 63   
 64      cus.cancel ()  # terminate cus 
 65      cps.cancel ()  # terminate compute pilots 
 66      ---------------------------------------------------------------- 
 67   
 68  The Data side of the Pilot API looks symmetric to the compute side.  The 
 69  combination of data and compute focused entities in the ComputeData-type classes 
 70  allows Troy to perform cross-cutting scheduling and optimization, prominently 
 71  compute-data co-scheduling. 
 72   
 73  """
