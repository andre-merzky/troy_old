  3  TROY: Tiered Resource Overlay 
  4  ============================= 
  5   
  6  Overview: 
  7  ========= 
  8   
  9    Troy is a scheduling overlay which specifically operates on pilot systems.  In 
 10    order to understand what Troy provides, it is helpful to discuss 'Scheduling' as 
 11    an abstract concept, and then apply it to scheduling on pilot resources. 
 12     
 13    We define the process of scheduling as follows:: 
 14     
 15        Assume a set of resources under control of a resource manager.  Users 
 16        communicate the intent to utilize resources by submitting description of said 
 17        utilization -- a workplan.  That workplan includes information about the 
 18        required subset of resources, the kind of work planned, and any additional 
 19        relevant or irrelevant information. 
 20       
 21        The resource manager can either decline to enact the requested workplan, or to 
 22        enact it at some point in time, on some subset of its resources.  That 
 23        (manager internal) decision making process is called scheduling.   
 24       
 25        The scheduling process can very well be (and usually is) decentral and 
 26        hierarchical:  a sequence of scheduling subprocesses is utilized to make the 
 27        final enactment decision and placement.  Any one of those scheduling 
 28        subprocesses may add additional constraints to the workplan, such as limit its 
 29        enactment to a specific part of the over all resource set, constrain the 
 30        co-locality of certain elements of the workplan, constrain the execution time 
 31        and accounting method for the workplan, etc.  If any of the scheduling 
 32        constraints contradict each other, or if the application of the constraints to 
 33        the resource set results in an empty set, the enactment of the workplan will 
 34        be impossible, and the scheduling request is declined [1].  Once the complete 
 35        set of constraints pinpoints the enactment of the workplan in space and time, 
 36        i.e. once the workplan has no free parameters, the workplan is considered to 
 37        be 'scheduled for execution'.   
 38       
 39        Note that a complex workplan may well be split apart into smaller work units, 
 40        which may get scheduled and enacted individually.  In those cases, a workplan 
 41        may become partially scheduled or enacted.  In particular, parts of the 
 42        original workplan may well become enacted while other parts are not yet 
 43        completely scheduled. 
 44     
 45     
 46    Troy provides means to implement one specific layer in the scheduling pipeline. 
 47    Specifically, Troy targets to support resources managed by pilot systems 
 48    (instances of pilot frameworks), and supports the application level scheduling 
 49    of workplans across those pilot systems (note that this Troy level scheduling 
 50    process may in turn consist of multiple subprocesses, i.e.  subschedulers). 
 51    Once the Troy level schedulers are done, the workplans are forwarded to the 
 52    underlying pilot systems for further scheduling and enactment.  Any scheduler on 
 53    Troy level may perform complete scheduling, i.e. schedule the workplan for 
 54    execution on a specific (set of) pilot(s); or may perform partial scheduling, 
 55    i.e. add a number of constraints which limit the scheduling options of the 
 56    downstream pilot framework schedulers.  Troy schedulers may split up or combine 
 57    workplans before forwarding them. 
 58     
 59    The L{troy.Troy} class is the user (or application) facing component of Troy. 
 60    Its purpose is to manage a set of pilot systems (and their pilot resources), and 
 61    to manage a set of scheduling subprocesses which perform scheduling over the 
 62    pilot resources, as described above.  As such, L{troy.Troy} has three distinct 
 63    sets of methods: to manage pilot resources, to manage scheduling subprocesses, 
 64    and to handle workplans to be scheduled on the pilot resources. 
 65     
 66     
 67  Terminology: 
 68  ============ 
 69     
 70    For the purpose of Troy, all backend pilot frameworks are expected to implement 
 71    the P* model of pilot abstractions [1], and to expose the Pilot API associated 
 72    with the P* model.  P* and Pilot API have the following terminology, which is 
 73    referenced throughout the Troy documentation:: 
 74     
 75        - ComputeUnitDescription: 
 76          a workplan which describes a set of planned computational tasks. 
 77     
 78        - DataUnitDescription 
 79          a workplan which describes a set of storage resource allocations (files, 
 80          data sets). 
 81     
 82        - ComputeUnit: 
 83          representation of an enacted (compute) workplan. 
 84     
 85        - DataUnit: 
 86          representation of an enacted (data) workplan. 
 87     
 88        - ComputePilotDescription: 
 89          a workplan which describes a ComputePilot. 
 90     
 91        - DataPilotDescription: 
 92          a workplan which describes a DataPilot. 
 93     
 94        - ComputePilot: 
 95          representation of a resource which can enact ComputeUnitDescriptions, 
 96          managed by a pilot. 
 97     
 98        - DataPilot: 
 99          representation of a resource which can enact DataUnitDescriptions, managed 
100          by a pilot. 
101     
102        - ComputePilotService: 
103          representation of a pilot manager which handles ComputePilots. 
104     
105        - DataPilotService: 
106          representation of a pilot manager which handles DataPilots. 
107     
108        - ComputeUnitService: 
109          representation of a Pilot system entity which schedules and enacts 
110          ComputeUnits over a set of ComputePilotServices. 
111     
112        - DataUnitService: 
113          representation of a Pilot system entity which schedules and enacts 
114          DataUnits over a set of DataPilotServices. 
115     
116        - ComputeDataUnitService: 
117          representation of a Pilot system entity which schedules and enacts 
118          ComputeDataUnits over a set of ComputeDataPilotServices. 
119     
120     
121    Troy's own terminology is very close to the P* and PilotAPI terminology, but 
122    as Troy has a different semantic and scope as P* systems, its terminology must 
123    necessarily differ:: 
124     
125        - Scheduler: 
126          Troy level manager of SchedulingAlgorithm instances. 
127     
128        - SchedulingAlgorithm: 
129          Troy level sub scheduler, manager by the troy.Scheduler. 
130     
131        - PilotFramework:  
132          representation of any pilot system, exposing the UnitService 
133          (DataUnitService, ComputeUnitService, and ComputeDataUnitService) and 
134          PilotService (DataPilotService and ComputePilotService) semantics. 
135     
136        - ComputePilotDescription: as in P* / Pilot API 
137        - DataPilotDescription:    as in P* / Pilot API 
138        - ComputePilot:            as in P* / Pilot API 
139        - DataPilot:               as in P* / Pilot API 
140        - ComputeUnitDescription:  as in P* / Pilot API 
141        - DataUnitDescription:     as in P* / Pilot API 
142        - ComputeUnit:             as in P* / Pilot API 
143        - DataUnit:                as in P* / Pilot API 
144     
145     
146    [1] http://radical.rutgers.edu/publications/#pstar12

149  Hello World: 
150  ============ 
151     
152    A simple Troy application scheduling a Compute centric workplan would be (on 
153    'Hello World' level, some details left out for brevity):: 
154     
155        ---------------------------------------------------------------- 
156        import troy 
157     
158        # create Tiered Resource Overlay ;-) 
159        t = troy.Troy () 
160     
161        # add a Round Robin scheduling algorithm 
162        s = troy.Scheduler ('round_robin') 
163        t.add_scheduler (s) 
164     
165        # create two ComputePilots from a BigJob PilotFramework 
166        pf = troy.PilotFramework ('bigjob://') 
167        pf.submit_pilot ([some compute pilot description]) 
168        pf.submit_pilot ([some compute pilot description]) 
169     
170        # add the bigjob PF to Troy, to submit work items to 
171        t.add_pilot_framework (pf) 
172     
173        # submit a compute work item 
174        cu = t.submit_unit ([some compute unit description]) 
175     
176        cu.wait ()     # wait 'til work is done 
177     
178        pf.cancel ()   # terminate compute pilots and bigjob 
179        t.cancel ()    # terminate Troy 
180        ---------------------------------------------------------------- 
181     
182    The Data side of the Troy looks symmetric to the compute side.  The combination 
183    of data and compute pilots and work items entities allows Troy to perform 
184    cross-cutting scheduling and optimization, prominently compute-data 
185    co-scheduling.
