PilotFramework (PF)

A PF acts as the interface to an underlying pilot job framework -- that backend creates and manages pilot instances (ComputePilot, DataPilot), and can schedule work unit amongst those pilots (ComputeUnit, DataUnit).

Class instances are identified by an url, and can be reconnected to. A PF can be added to a Troy instance, which will then be able to utilize the PF's pilots for work unit scheduling and execution.

Properties:

   - id:
     The returned ID can be used to connect to the PF instance later on,
     for example from within a different application instance.
     type: string (url)

Create a PilotFramework object

Keyword arguments: url: url identifying the backend PF.

Note that the URL may be incomplete, if a new PF instance is to be created -- for example, it may contain only a hint about what pilot framework is to be used on what resource. On inspection, the PF will always return a fully qualified URL, which will not change over the lifetime of the PF.

submit_pilot(self, pd)

source code 

Create a Pilot.

Keyword arguments: pd -- PilotDescription

Return value: A ComputePilot or a DataPilot instance, depending on type of pilot description.

If the resource requirements defined in the pd cannot be met by the PF, a BadParameter exception is raised. Not raising this exception is, however, not a guarantee that the pilot will in fact be (able to be) executed -- in that case, the returned pilot will be moved to Failed state.

On success, the returned pilot is in Pending state (or moved into any state downstream from Pending).

submit_pilot() will honor all attributes set on the PD. Attributes which are not explicitly set are interpreted as having default values (see documentation of ComputePilotDescription and DataPilotDescription), or, where default values are not specified, are ignored.

Note: compared to the Pilot API, this PF combines the functionality of the UnitServices and the PilotServices. Any backend level decomposition is considered a PF implementation detail, and, if needed, used only on adaptor level.

list_pilots(self)

source code 

List managed pilots.

Return value: A list of ComputePilot and DataPilot IDs.

The returned list can include pilots which have not been created by this PF instance. The list may be incomplete, and may not include pilots created by the PF. There is no guarantee that pilots in the returned list can in fact be reconnected to. Also, an inclusion in the list does not have any indication about the respective pilot's state.

submit_unit(self, ud)

source code 

Submit a work unit to this PilotFramework.

Keyword argument: ud -- ComputeUnitDescription or DataUnitDescription to be enacted.

Return: ComputeUnit or DataUnit instance, depending on type of description.

The UD is passed on to the PF backend, which will attempt to instantiate the described workload on any of its pilots. If no suitable pilot is found, a Error.BadParameter exception is raised. Not raising this exception is not a guarantee that the work unit will in fact be (able to be) executed -- in that case, the returned work unit will later be moved to Failed state.

On success, the returned work unit is in Pending state (or moved into any state downstream from Pending).

The call will will honor all attributes set on the UD. Attributes which are not explicitly set are interpreted as having default values (see documentation of UD), or, where default values are not specified, are ignored.
